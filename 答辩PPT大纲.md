## 第三页：研究背景与动机（约1min）

**背景数据**
- 截至2025年底，全国汽车保有量达3.66亿辆，连续11年新注册超300万辆
- 城市停车难、车位利用率低、进出口效率低下问题突出

**传统停车场痛点**
- 需人工值守，人力成本高
- 信息孤岛，无法联网管理
- 人工计时误差大，易产生纠纷

**现有嵌入式方案的三大不足**

| 维度 | 问题描述 |
|------|---------|
| 通信实时性 | 主控同时负责设备控制与网络通信，资源竞争导致响应迟缓 |
| 识别容错性 | 单帧识别无容错手段，弱光/运动模糊下错误率高 |
| 车辆管控 | 无法区分车辆类型，逃费/欠费车辆无法有效拦截 |

## 第五页：改进点一 —— 双MCU异步通信架构（约0.7min）

**问题**：传统AT指令模式同步阻塞，40KB图片上传期间主控被阻塞约6250ms，传感器无法响应

**本文方案**：
- ESP8266烧录自定义Arduino固件，作为独立网络协处理器
- STM32与ESP8266通过UART以**换行符分隔JSON**异步通信
- 主控发出指令后立即返回，无需等待网络操作完成
- 支持运行时动态配置WiFi/Broker地址，无需重烧固件

**效果预览**（数字留到实验页展示）：主控阻塞时间 = **0**

---

## 第六页：改进点二 —— 双帧比对容错识别（约0.7min）

**问题**：OV7670成像质量有限，单帧易受运动模糊和光线变化影响，无容错手段

**本文方案**：触发后间隔1500ms连续采集两帧，上传至服务端在5s窗口内执行融合决策

**五级融合规则**（放图5.3流程图）：

| 规则 | 触发条件 | 决策动作 |
|------|---------|---------|
| 规则一 | 两帧结果完全一致 | 取首帧，直接采信 |
| 规则二 | 仅次帧通过格式校验 | 取次帧 |
| 规则三 | 仅首帧通过格式校验 | 取首帧 |
| 规则四 | 两帧均有效但字符不同 | 编辑距离择优 |
| 规则五 | 两帧均未通过格式校验 | 保守取首帧，记日志 |

## 第七页：改进点三 —— 三级入场决策机制（约0.7min）

**问题**：现有低成本方案无法区分车辆类型，逃费/欠费车辆无法被拦截

**本文方案**（放图5.4车辆入场业务流程图）：

```
车牌识别完成
    ↓
第一级：黑名单？ → 是 → 拒绝放行 + MQTT下发DENY指令 + 写操作日志
    ↓ 否
第二级：白名单/会员？ → 是 → 优惠放行，标注优惠类型
    ↓ 否
第三级：普通计费放行 → 创建停车记录 → MQTT下发开闸 → 红外确认进入 → 开始计费
```

决策结果通过MQTT一次性下发，无需轮询。

**嵌入式端**

## 第十页：工程联调中的完善（约1min）

> 本页展示论文测试完成后、系统集成联调阶段发现并解决的两个问题，体现从方案设计到工程落地的完整过程。

### 一、黑名单拦截：从"依赖时序"到"双重保障"

**问题根因**：设备端开闸标志 `gate_allow` 硬编码为 `1`，嵌入式端未解析 HTTP 响应中服务端下发的 `allow` 字段。论文阶段拦截测试通过，是因为 MQTT `deny` 指令恰好先于开闸动作到达——若 MQTT 存在延迟，黑名单车辆仍会被放行。

**修复方案**：解析 HTTP 响应中的 `allow` 字段，改为 `gate_allow = best_plate.allow`。

**效果**：HTTP响应 `allow=0` 与 MQTT `deny` 指令**双重保障**，黑名单拦截在任何时序下均可靠生效。

---

### 二、YL62 监测：从"主任务阻塞"到"后台独立线程"

**问题根因**：论文方案中开闸后主任务需在循环内持续轮询 YL62 等待车辆到位，期间主任务阻塞约 3s，无法处理其他事务。

**改进方案**：将 YL62 提取为**独立后台线程**（50ms 轮询），开机即持续运行，主任务调用 `YL62_Monitor_IsObstacle()` 直接读取已积累的实时状态。

**效果**：消除开闸阶段主任务阻塞，状态变化自动通过 MQTT 上报，主任务与 YL62 监测完全解耦。

## 第十一页：总结与展望（约0.5min）

**研究结论**

| 改进点 | 方案 | 核心指标 |
|-------|------|---------|
| 通信实时性 | 双MCU异步JSON协议 | 40KB延迟降低83.8%，主控阻塞=0 |
| 识别容错性 | 双帧比对五级融合 | 整体准确率提升9.5%，恶劣环境最高提升15% |
| 车辆管控 | 三级入场决策引擎 | 黑/白/普通三类车辆差异化管控，集成测试全通过 |

**工程联调补充**：黑名单双重保障闭环 ｜ YL62 后台监测消除主任务阻塞

**不足与展望**
- 摄像头成像质量限制识别上限 → 可升级支持自动对焦高清摄像头
- 推理依赖网络传输 → 探索边缘端轻量化模型离线推理
- 网络断开时无法计费 → 增加本地离线缓存与批量同步机制
- 未覆盖场内车位引导 → 结合超声波矩阵实现车位级精准管理

> **建议准备的高频答辩问题**：
> 1. 为何不用ESP32单芯片方案？
>    → ESP32无硬件JPEG编码器，软件编码慢；WiFi协议栈持续占用CPU，影响传感器实时控制
> 2. 双帧方案时延代价如何取舍？
>    → 单帧≤2s，双帧≤5s，以约2s时延换来9.5%精度提升，对停车场场景可接受
> 3. 97%上传成功率的3次失败原因？
>    → 均发生在设备冷启动网络握手阶段，连接稳定后未再出现
> 4. 黑名单集成测试已通过，为何联调才发现问题？
>    → TC-02通过是因为MQTT deny指令在时序上先于闸门动作，硬编码问题被正常时序掩盖；联调中模拟MQTT延迟场景才暴露，修复后HTTP响应与MQTT指令双重保障
> 5. YL62后台线程与主任务共享数据，如何保证一致性？
>    → volatile修饰共享变量，getter为单次对齐读，Cortex-M7天然原子操作，无需互斥锁
